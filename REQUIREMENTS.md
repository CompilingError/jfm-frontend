# jfm-frontend 需求文档

版本：v0.2
更新日期：2026-06-02

## 1. 项目定位

`jfm-frontend` 是 `jfm-backend` 的桌面前端应用，面向没有前端基础的开发学习过程，采用“Step by step, part by part”的方式逐步实现。

最终用户只需要启动 Windows `.exe` 程序，就可以在本地独立窗口中完成文件管理、文件夹监视、新增文件审查、列表筛选、标签与艺人维护等操作。

## 2. 技术方案

### 2.1 前端技术

- 使用 JavaScript React 开发界面。
- 推荐使用 Vite 创建 React 项目。
- 页面组件应按功能拆分，避免一开始写成单个巨大页面。

### 2.2 桌面应用技术

- 使用 Electron 打包整个应用。
- Electron 负责创建本地窗口、选择文件夹、监视文件夹、与操作系统文件系统交互。
- React 只负责界面和业务交互，不直接访问 Node.js 文件系统 API。

### 2.3 后端集成

- 前端通过 HTTP 调用本地 Spring Boot 后端 API。
- 默认后端地址为 `http://localhost:8080`。
- 后续打包时需要考虑 Electron 应用、React 静态资源、Spring Boot jar、Java Runtime 的集成方式。

## 3. 核心用户流程

### 3.1 第一次使用流程

1. 用户启动 `jfm-frontend.exe`。
2. 应用打开 Electron 桌面窗口。
3. 应用连接或启动本地后端服务。
4. 用户进入“监视文件夹设置”。
5. 用户点击“添加文件夹”。
6. Electron 打开系统文件夹选择窗口。
7. 用户选择一个或多个文件夹。
8. 应用开始监视这些文件夹。
9. 新文件出现后进入“待审查”列表。
10. 用户审查文件信息。
11. 审查通过后，文件进入正式列表。

### 3.2 日常使用流程

1. 用户启动应用。
2. 应用恢复上次保存的监视文件夹配置。
3. 用户进入文件列表页。
4. 用户通过 filter 条件筛选文件。
5. 新增文件出现时，应用提醒用户审查。
6. 用户维护标签、艺人、描述、收藏状态等信息。

## 4. MVP 功能范围

第一阶段优先实现：

- Electron 启动 React 页面。
- React 连接本地后端。
- 文件列表页。
- 重点支持列表 filter 功能。
- 标签管理。
- 艺人管理。
- 文件夹选择。
- 文件夹监视。
- 新增文件进入待审查列表。
- 用户审查后导入后端。

第一阶段暂不实现：

- 自动识别影片元数据。
- 内置播放器。
- 云同步。
- 用户登录。
- 文件真实删除。
- 基于文件内容 hash 的完整去重体系。

## 5. 文件列表功能

### 5.1 列表目标

文件列表是前端核心页面之一。它不仅用于展示文件，还要重点支持 filter 功能，帮助用户快速定位文件。

列表默认展示后端 `/movies` 返回的数据。

### 5.2 基础展示字段

文件列表至少展示：

| 字段 | 说明 |
| --- | --- |
| 名称 | 文件名称或用户自定义名称 |
| 路径 | 当前文件路径 |
| 标签 | 文件关联标签 |
| 艺人 | 文件关联艺人 |
| 喜欢 | 是否收藏 |
| 新鲜度 | `freshVal` 值 |
| 文件状态 | 正常、缺失、待审查等状态，后续扩展 |
| 操作 | 查看、编辑、删除记录、标记已观看 |

### 5.3 分页需求

列表需要支持分页：

- 当前页。
- 每页数量。
- 总数量。
- 上一页。
- 下一页。

示例请求：

```http
GET /movies?page=0&size=20
```

### 5.4 Filter 需求

列表功能的重点是 filter，而不是只做简单展示。

第一阶段建议支持以下筛选条件：

| Filter 条件 | 说明 | MVP 是否实现 |
| --- | --- | --- |
| 文件名称 | 按名称关键字搜索 | 是 |
| 标签 | 按一个或多个标签筛选 | 是 |
| 艺人 | 按一个或多个艺人筛选 | 是 |
| 是否喜欢 | 只看收藏或全部 | 是 |
| 文件状态 | 正常、缺失、待审查等 | 后续 |
| freshVal 范围 | 按新鲜度区间筛选 | 后续 |
| 文件扩展名 | 如 `.mp4`、`.mkv` | 后续 |
| 监视文件夹 | 只看某个文件夹下的文件 | 后续 |
| 创建时间 | 按入库时间筛选 | 后续 |
| 更新时间 | 按最近修改时间筛选 | 后续 |

### 5.5 Filter 交互设计

列表页顶部建议设计 filter bar：

```text
┌────────────────────────────────────────────────────┐
│ 搜索名称  标签多选  艺人多选  喜欢状态  重置筛选    │
└────────────────────────────────────────────────────┘
```

用户修改 filter 后，页面应刷新列表。

建议行为：

- 名称搜索可以延迟触发，避免每输入一个字符都请求后端。
- 标签和艺人使用多选组件。
- 筛选条件应显示当前生效状态。
- 提供“一键清空筛选”。
- 筛选条件变化时，分页应回到第一页。

### 5.6 Filter 与后端 API 的关系

当前后端已有这些相关接口：

```http
GET /movies
GET /movies/name/{name}
GET /movies/by-tags?tagIds=1&tagIds=2&mode=ALL
GET /movies/by-artists?artistIds=1&artistIds=2&mode=ANY
```

MVP 阶段可以先使用现有接口完成基础筛选。

后续为了支持组合 filter，建议后端新增统一查询接口，例如：

```http
GET /movies/search?name=abc&tagIds=1,2&artistIds=3&like=true&page=0&size=20
```

或者使用 POST 查询：

```http
POST /movies/search
```

请求体示例：

```json
{
  "name": "abc",
  "tagIds": [1, 2],
  "tagMode": "ALL",
  "artistIds": [3],
  "artistMode": "ANY",
  "like": true,
  "page": 0,
  "size": 20
}
```

## 6. 文件夹监视功能

### 6.1 添加监视文件夹

用户点击“添加监视文件夹”后，Electron 打开系统文件夹选择窗口。

用户可以选择一个或多个文件夹。

应用保存这些文件夹路径，并在下次启动时自动恢复。

### 6.2 监视新增文件

当监视文件夹中出现新文件时，系统应：

1. 检测新增文件路径。
2. 判断该文件是否已经被忽略或已经导入。
3. 将未处理的新文件加入“待审查”列表。
4. 提示用户有新文件需要审查。

### 6.3 监视删除文件

当监视文件夹中的文件被删除时，第一阶段不直接删除数据库记录。

建议行为：

- 前端提示文件缺失。
- 后续在后端增加文件状态字段，例如 `AVAILABLE`、`MISSING`。
- 用户可以手动确认是否删除数据库记录。

## 7. 新增文件审查功能

### 7.1 待审查列表

新增文件必须先进入“待审查”列表，不应直接进入正式文件列表。

待审查列表字段：

| 字段 | 说明 |
| --- | --- |
| 文件名 | 默认取本地文件名 |
| 路径 | 当前检测到的路径 |
| 检测时间 | 文件进入待审查列表的时间 |
| 状态 | 待审查、已忽略等 |
| 操作 | 审查、忽略、稍后处理 |

### 7.2 审查弹窗

用户点击“审查”后，弹窗展示：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 名称 | 是 | 默认使用文件名 |
| 路径 | 是 | 自动填入，第一阶段可只读 |
| 描述 | 否 | 用户手动填写 |
| 标签 | 否 | 多选 |
| 艺人 | 否 | 多选 |
| 是否喜欢 | 否 | 默认 false |

### 7.3 审查结果

用户可以选择：

- 保存并导入。
- 忽略。
- 稍后处理。
- 取消。

保存并导入时调用：

```http
POST /movies
```

请求体示例：

```json
{
  "name": "example",
  "path": "D:/Movies/example.mp4",
  "description": "用户填写的描述",
  "tagIds": [1, 2],
  "artistIds": [3]
}
```

## 8. 文件唯一性与 hashValue 规划

### 8.1 当前问题

第一阶段可以临时使用 `path` 来识别文件，但 `path` 并不是稳定标识。

原因：

- 文件可以被移动。
- 文件可以被重命名。
- 不同路径可能指向相同内容的文件副本。
- 仅依赖 path 容易出现重复导入。

因此，后续需要使用基于文件本身内容解析得到的 `hashValue` 来替代 path 作为更稳定的文件身份标识。

### 8.2 hashValue 后期目标

后续功能中，系统应读取文件内容并计算 hashValue，例如 SHA-256。

`hashValue` 的目标用途：

- 判断两个文件内容是否相同。
- 避免重复导入 duplicate 文件。
- 支持文件移动或重命名后的重新识别。
- 降低 path 变化对数据稳定性的影响。

### 8.3 path 与 hashValue 的职责划分

后续设计中：

| 字段 | 职责 |
| --- | --- |
| `hashValue` | 文件内容身份标识，用于去重和识别同一文件 |
| `path` | 文件当前位置，用于打开文件和展示路径 |
| `name` | 用户可编辑的显示名称 |

也就是说，`hashValue` 用于回答：

```text
这是不是同一个文件？
```

`path` 用于回答：

```text
这个文件现在在哪里？
```

### 8.4 hashValue 阶段性实现建议

第一阶段不强制实现完整 hash 去重，避免增加学习复杂度。

建议分阶段：

1. MVP 阶段：使用 path 作为临时识别依据。
2. 第二阶段：Electron 在发现新增文件时计算 hashValue。
3. 第三阶段：后端保存 hashValue，并用 hashValue 做唯一约束。
4. 第四阶段：如果发现相同 hashValue 但路径不同，则提示用户这是重复文件或移动后的文件。
5. 第五阶段：提供“更新路径”“保留两个路径”“忽略重复文件”等处理方式。

### 8.5 对后端的后续影响

后续建议后端将文件唯一性从 path 转向 hashValue。

当前数据库实体中可以保留 path，但唯一性应逐步由 hashValue 承担。

建议后端字段：

```text
hashValue: string, required, unique
path: string, required, not unique or conditionally unique
```

如果考虑一个相同文件存在多个路径副本，则可以进一步设计为：

```text
files
- id
- hashValue
- name
- description

file_locations
- id
- fileId
- path
- exists
```

这个模型更完整，但不建议在 MVP 阶段实现。

## 9. 标签管理

标签页面需要支持：

- 查看标签。
- 新建标签。
- 编辑标签。
- 删除标签。

对应接口：

```http
GET /tags
POST /tags
PUT /tags/id/{id}
DELETE /tags?ids=1&ids=2
```

## 10. 艺人管理

艺人页面需要支持：

- 查看艺人。
- 新建艺人。
- 编辑艺人。
- 删除艺人。

对应接口：

```http
GET /artists
POST /artists
PUT /artists/id/{id}
DELETE /artists?ids=1&ids=2
```

## 11. 推荐开发顺序

建议按以下顺序教学式开发：

1. 创建 Vite React 项目。
2. 引入 Electron。
3. 理解 Electron main process、preload、renderer。
4. React 连接后端。
5. 实现主布局。
6. 实现文件列表页。
7. 在文件列表页重点实现 filter bar。
8. 实现标签管理。
9. 实现艺人管理。
10. 实现文件夹选择。
11. 实现文件夹监视。
12. 实现待审查列表。
13. 实现审查弹窗。
14. 打包为 Windows `.exe`。
15. 后续加入 hashValue 文件唯一性识别。

## 12. 验收标准

### 12.1 MVP 验收

MVP 完成时应满足：

- 双击 `.exe` 可以打开本地桌面窗口。
- 应用可以连接本地后端。
- 文件列表可以正常加载。
- 文件列表可以按名称、标签、艺人、喜欢状态进行基础 filter。
- 可以添加监视文件夹。
- 新增文件会进入待审查列表。
- 用户审查后可以导入正式列表。
- 可以维护标签。
- 可以维护艺人。

### 12.2 后续 hashValue 验收

hashValue 功能完成后应满足：

- 同一内容文件即使路径不同，也能识别为重复文件。
- 文件重命名后，不会被错误当成全新文件重复导入。
- path 改变不会破坏文件本身的唯一身份。
- 用户可以处理重复文件提示。
